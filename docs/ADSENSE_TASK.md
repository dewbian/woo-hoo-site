# woo-hoo.kr 애드센스 통과 작업 지시서

> **목표**: `woo-hoo.kr`(메인 도메인) 기준으로 Google AdSense 승인받기.
> **전략(A안)**: 자동 뉴스 블로그(`blog.woo-hoo.kr`)는 심사에서 분리하고, 메인의 **devlog(개발기)** 를 고유 콘텐츠 자산으로 키워 신청한다.
> **거절 사유**: "가치가 별로 없는 콘텐츠(thin content)". 따라서 **읽을 가치가 있는 고유 글**과 **신뢰성 페이지(About/Privacy/Contact)** 확보가 핵심.

---

## ⚠️ 절대 원칙 (작업 전 반드시 인지)

1. **광고 코드는 `woo-hoo.kr` 에만 붙인다.** `blog.woo-hoo.kr`(자동 뉴스)에는 절대 붙이지 않는다. 자동 양산 콘텐츠에 광고를 붙이면 계정 단위 정책 위반(scaled content abuse)으로 번질 수 있음.
2. **devlog 글은 사람이 직접 쓴 고유 경험이어야 한다.** AI로 본문을 양산하지 않는다. (블로그가 거절될 결정적 사유였음 — LLM 프롬프트 누수, 뉴스 짜깁기, 페르소나 중복 발행)
3. **심사 기간 동안 메인 → 블로그 링크 노출을 최소화한다.**

---

## 작업 순서 (우선순위)

```
PHASE 1: 신뢰성/구조 정비  (링크 정리 + 필수 페이지)
PHASE 2: devlog 구조화      (개별 URL 페이지로 분리)
PHASE 3: 콘텐츠 채우기      (글 12~15개 — 사람이 작성)
PHASE 4: 최종 점검 후 신청
```

---

## PHASE 1 — 신뢰성/구조 정비

### TASK 1-1. 깨진 링크 전부 수정
현재 메인(`index`)에 `href="#"` 로 비어 있는 링크들이 있음. 심사관이 "미완성 사이트"로 판단하는 직접 사유.

- [ ] Play 스토어 링크(`self-check-study`의 "Play 스토어 열기 ↗") — 실제 스토어 URL 연결. 미출시면 **버튼 자체를 제거**하거나 "출시 예정" 비활성 상태로 표기.
- [ ] 푸터 `GitHub ↗` (`href="#"`) — 실제 GitHub 프로필 연결 or 제거
- [ ] 푸터 `X / Twitter ↗` (`href="#"`) — 실제 계정 연결 or 제거
- [ ] devlog 카드의 "개발 로그 보기 →", "기능 명세 보기 →" 등 `#` 링크 — PHASE 2에서 만들 실제 페이지로 연결

> **규칙**: 채울 수 없는 링크는 남겨두지 말고 제거한다. 빈 `#` 링크가 thin content 인상을 키운다.

### TASK 1-2. About 페이지 (`/about` 또는 `/about.html`)
현재 `#about` 앵커만 존재. 독립 페이지로 제작.

포함 내용:
- [ ] 운영자 소개 (실재하는 1인 개발자임이 드러나게 — 닉네임이라도 일관되게)
- [ ] 이 실험실을 왜/언제부터 운영하는지
- [ ] 무엇을 만들고 어떻게 공개하는지 (운영 철학)
- [ ] 연락처 (이메일)
- [ ] 메인 헤더 네비에 `About` → 이 페이지로 연결

### TASK 1-3. Contact 정보 통일
현재 이메일이 두 개로 갈림: `woohoo.dew1017@gmail.com`(메인) / `dev@b-lot.co.kr`(블로그).

- [ ] 대표 이메일 **하나로 통일**
- [ ] About 페이지 또는 별도 `/contact` 에 명확히 노출

### TASK 1-4. Privacy Policy — 애드센스 문구 추가 (필수)
현재 `/privacy/` 존재하지만 광고 관련 문구가 없을 가능성. 아래 항목이 반드시 포함되어야 함:

- [ ] 제3자 광고 공급자(Google)가 쿠키를 사용해 광고를 게재한다는 고지
- [ ] Google의 DoubleClick 쿠키 사용 및 사용자가 광고 설정(adssettings.google.com)에서 비활성화 가능하다는 안내
- [ ] 수집 정보 범위 / 쿠키 사용 / 제3자 링크 면책
- [ ] 문의 연락처

> 표준 문구 초안이 필요하면 별도로 요청. (한국어 + 애드센스 권고 문구 기준)

### TASK 1-5. (선택) 이용약관 / Terms
- [ ] 필수는 아니나 신뢰도 가산. 간단한 면책/저작권 조항 페이지 추가 권장.

### TASK 1-6. 심사 기간 한정 — 블로그 링크 격리
- [ ] 메인 헤더 네비의 `세상만사 ↗`(`https://blog.woo-hoo.kr`) 링크를 **제거** 하거나 `rel="nofollow"` 처리
- [ ] 푸터의 블로그 링크도 동일 처리
- [ ] (승인 후 원복 가능)

## PHASE 2 — devlog 구조화

### TASK 2-1. devlog를 개별 URL 페이지로 분리
현재 devlog 글들이 메인 페이지 `#devlog` 앵커 안에 묻혀 있음 → 실질 페이지 수로 잡히지 않음.

- [ ] 글마다 독립 URL 부여: 예) `/devlog/stamp-interaction`, `/devlog/persona-pipeline`
- [ ] devlog 인덱스 페이지(`/devlog`)에서 각 글로 링크
- [ ] 각 글 페이지에 제목 / 작성일 / 본문 / 태그 / 이전·다음 글 링크 구조
- [ ] 기존 메모 3개(도장 인터랙션 / 페르소나 5종 / googSky 안정성 점수)를 1순위로 개별 페이지화

### TASK 2-2. 내부 링크 연결
- [ ] 각 devlog 글에서 관련 프로젝트 섹션(self-check-study 등)으로 내부 링크
- [ ] 글 하단에 "다른 개발기 보기" 리스트

## PHASE 3 — 콘텐츠 채우기 (사람이 직접 작성)

> 목표: **완성글 12~15개**. 각 글 1,000자 이상.
> 공통 구조: **문제 상황 → 시도 → 막힌 지점 → 해결(or 실패) → 배운 것**
> 실패담 · 스크린샷 · 코드 조각이 들어갈수록 강함.

### self-check-study
- [ ] 1. 학생 5명 유저테스트에서 배운 것 — "한 번 더 하고 싶다"를 만든 0.4초
- [ ] 2. 체크박스를 도장으로 바꾼 이유 — 성취감의 UX 설계
- [ ] 3. 카톡 공유 카드 OG 이미지 튜닝 삽질기 (사이즈/캐싱)
- [ ] 4. Play 스토어 심사 통과까지 — 개인정보처리방침·권한 정리

### make-money
- [ ] 5. 뉴스 크롤러와 robots.txt / 차단 이슈
- [ ] 6. 티스토리 OAuth 토큰 만료 자동 복구 흐름
- [ ] 7. LLM 프롬프트 체인 안정화 — 같은 입력에 다른 출력 줄이기
- [ ] 8. 자동발행 큐를 30분 간격으로 분산한 이유
       (↳ "AI 양산이 애드센스에서 왜 위험한가"를 경험담으로 — 메타적으로 좋은 글)

### googSky
- [ ] 9. 입시 등급 데이터 정규화 — 학교별 백분위 칼럼 통일
- [ ] 10. "안정·적정·도전"을 안정성 점수로 재정의한 과정
- [ ] 11. 학부모 인터뷰 8건에서 가장 많이 나온 질문과 거기서 바꾼 것

### 공통 / 회고
- [ ] 12. 1인 개발자가 앱 3개를 동시에 굴리는 법 — 우선순위 관리
- [ ] 13. Claude로 코드 짜면서 좋았던 것 / 안 통한 것
- [ ] 14. woo-hoo.kr v2 리디자인 — 왜 "완성된 회사 사이트"를 버렸나
- [ ] 15. 사이드프로젝트 인프라 짜기 — 도메인·배포·비용

## PHASE 4 — 최종 점검 후 신청

### 제출 전 체크리스트
- [ ] devlog 완성글 12개 이상, 각 1,000자 이상, 개별 URL
- [ ] About / Privacy(광고문구 포함) / Contact 페이지 정상 동작
- [ ] 메인의 모든 `#` 빈 링크 제거 또는 실제 연결 완료
- [ ] 모바일 레이아웃 깨짐 없음 (반응형 확인)
- [ ] 메인 → 블로그 링크 격리(nofollow/제거) 적용
- [ ] `blog.woo-hoo.kr` 에 광고 코드 없음 확인
- [ ] 404/깨진 페이지 없음, sitemap.xml 정상
- [ ] AdSense 코드는 `woo-hoo.kr` 의 `<head>` 에만 설치

### 신청
- [ ] AdSense 콘솔에서 **`woo-hoo.kr`** 사이트 추가 → 코드 설치 확인 → 검토 요청
